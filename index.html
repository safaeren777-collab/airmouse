The Problem

Why does mid-air cursor control fail?

The first three versions tried the classic approach: map hand position to a cursor and bind static hand poses (open palm, fist, peace sign) to commands. Two structural problems emerged:

🦍 Gorilla Arm Syndrome

Precise movements with an arm held in the air become exhausting within minutes. The fix, well documented in HCI research, is not bigger gestures but small, supported movements performed from a resting position.

📳 Gesture Flicker

Static pose classification changes frame to frame. With a single threshold, an "open palm" command keeps opening and cancelling — V3 logs showed repeated activated → cancelled loops within the same second. The fix: hysteresis + a state machine.

Machine Learning

How the ML is actually used

AirMouse runs two on-device neural networks per frame through Google's MediaPipe Tasks API — no cloud, no GPU required (GPU delegate is attempted, with automatic CPU fallback). The interesting part is what sits on top of them.

MODEL 1 · HAND LANDMARKER

21 keypoints, two-stage CNN

A single-shot palm detector network localizes the hand, then a landmark regression network predicts 21 3D keypoints (fingertips, joints, wrist). Running in VIDEO mode exploits temporal coherence: the expensive detector only re-runs when tracking confidence drops, which keeps the whole pipeline at ~30 FPS on a laptop CPU.

MODEL 2 · FACE LANDMARKER

478-point mesh + 52 blendshapes

Regresses a dense 3D face mesh plus 52 blendshape coefficients — ARKit-compatible expression weights such as browInnerUp, jawOpen, eyeBlinkLeft — each a continuous 0–1 score of how strongly the user performs that expression.

DESIGN CHOICE

Geometry over classifiers

We deliberately did not train an end-to-end gesture classifier. Raw landmarks feed hand-engineered features: thumb-to-fingertip distance normalized by hand scale (wrist → middle-finger MCP), making detection invariant to hand size and camera distance. No training data, fully debuggable — every trigger is a number crossing a threshold.

NOISE → EVENTS

Hysteresis state machine

Per-frame neural predictions are noisy. A hysteresis band (pinch engages below ratio 0.40, releases only above 0.60) plus a state machine (IDLE → PINCHED → TAP / SCROLL / CAROUSEL / SWIPE) converts that noise into clean, discrete events. This is what fixed V3's flicker.

FALSE-POSITIVE FILTERS

Temporal validation

Blendshapes pass threshold + 250 ms minimum hold (a natural blink lasts ~150 ms, so blinks never fire) + a refractory period + cross-checks: a wink only counts if the other eye is open. Pinch midpoints are EMA-smoothed before drag distances are measured; a fist is rejected when two fingertips are both in pinch range.

# The core of hysteresis-based pinch detection — gesture_engine.py
hand_scale = dist(wrist, middle_mcp)              # normalize by hand size
ratio = dist(thumb_tip, finger_tip) / hand_scale

if ratio < 0.40:  # engage threshold  → pinch STARTS
    ...
if ratio > 0.60:  # release threshold → pinch ENDS (the gap = hysteresis)
    resolve tap / swipe / carousel close
The Solution

Pinch: a touch-down moment in mid-air

V4 brings the model Apple Vision Pro also adopted to the webcam: no cursor, only discrete commands. The primitive is the pinch (thumb + fingertip contact) — a single distance measurement that detects with extreme reliability.

01 · TOUCHPAD METAPHOR

Pinch = touch

What makes a laptop touchpad comfortable is that touching marks the start/end of every gesture. The pinch plays the same role in the air: pinch → drag → release.

02 · TAP / HOLD / DRAG

Three events from one gesture

Short pinch = tap (shortcut). Pinch + drag = a mode (scroll, carousel, swipe). Which finger pinches selects the command palette: 4 fingers = 4 separate "buttons".

03 · ALT-TAB CAROUSEL

Hold, sweep, release

Middle pinch + horizontal drag physically holds Alt; sweeping left/right walks through windows, releasing the pinch selects one. A guaranteed release_all() on every exit path means Alt can never get stuck.

04 · FULL FIELD OF VIEW

The 4:3 crop trap

Forcing 640x480 (4:3) makes 16:9 webcam sensors crop the sides of the image. The camera is opened at MJPG 1280x720 and downscaled to 640px for processing — full FOV at the same CPU cost.

05 · THREADED PIPELINE

Signals, not blocking

Vision runs in its own QThread; gesture events cross to the main thread as Qt signals where input simulation happens. The camera loop is never blocked by the GUI or the keyboard simulator.

06 · FACE MODE

Completely hands-free

Blendshape coefficients map to shortcuts — raise eyebrows for Alt-Tab, open mouth for Play/Pause, wink for Back/Forward. Works even while both hands stay on the keyboard. Toggled separately with Ctrl+Shift+Space.

Architecture

Signal-driven, multi-threaded design

Vision processing runs in a dedicated QThread; gesture events travel to the main thread as Qt signals, where input simulation happens. The camera loop is never blocked by the GUI or keyboard simulation.

Retrospective

The honest conclusion: what did the experiment teach?

Hypothesis: Can webcam gestures be an ergonomic alternative to the keyboard and touchpad on a laptop?

Answer: No — and the reason is physics, not engineering. A keyboard shortcut completes in ~200 ms under every condition. The chain of camera latency + model inference + lighting dependency only approaches that bar on its best day and loses consistency in bad light. Lifting a hand off the keyboard to gesture costs more effort than pressing the shortcut — so for an able-bodied user, the value proposition turns negative.

The real home of this technology is accessibility. As Google's Project Gameface also targets, these systems are not an alternative to the keyboard — they are a keyboard for those who cannot use one. V4's face mode (blendshape → shortcut) is genuinely meaningful in that scenario.

What carries forward: designing hysteresis-based gesture state machines, integrating the MediaPipe Tasks API (hand + face concurrently), Win32 input simulation and global hotkey management, click-through overlay windows, camera FOV negotiation (the 4:3 crop trap) — and above all, the experience of validating an HCI idea against research literature and testing it systematically across four versions.
